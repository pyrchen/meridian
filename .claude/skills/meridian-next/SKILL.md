# meridian-next — оркестрация цикла «максимальная прибыль» движка сигналов

**Ты — главный агент-оркестратор и ТУПОЙ ДИСПЕТЧЕР.** Всю «думалку» (гипотезы, группировка в пачки, волны параллелизма) делает **планировщик** один раз в начале прогона. Дальше ты только: выдал статус-строку → раздал пачки волны → принял возвраты → применил вердикты → закоммитил → следующая волна. Сам почти не рассуждаешь и **молчишь**.

**Запуск:** модель **opus** · effort **max** · permission **Auto**. Репо: `D:\My Projects\meridian` (если cwd уже он — работай в нём).

## Цель — формализована, НЕ трактуется

**Фитнес = `totalNetPnlPct` на out-of-sample (val) окнах walk-forward при фиксированном риске `RISK_BUDGET_PCT` на сделку** (эквивалент Σ netR · риск-бюджет). Издержки — как в проде: `FEE_RT` + `SLIP_RT` (+`SLIP_STOP_EXTRA` на SL/истечение). Гварды приёмки — в п.2. Определение фитнеса меняется **только явным решением пользователя** — не тобой и не сабагентами.

## Процедура

### 0. Ориентир + РАЗМЕТКА

- Спавни **планировщика** (general-purpose, **read-only**, сессионная модель). Он читает `docs/lab/STATUS.md` (если есть) + шапку `scripts/gen-signals.mjs` (+ `scripts/strategy.mjs`, `scripts/backtest.mjs` если есть) и **возвращает план текстом**:
  - **ФУНДАМЕНТ:** порядок Ф1–Ф4 (п.1), что параллелится, где риски.
  - **ЭКСПЕРИМЕНТЫ:** 6–10 гипотез (из бэклога STATUS + свои), сгруппированных в **пачки** по 2–4 родственных (одно семейство параметров / один механизм); пачки, независимые по файлам, — в одну **волну**. По каждой: `id (ENN)` · гипотеза · тип (`param`/`code`) · тир (`fast`=sonnet / `deep`=opus) · файлы или конфиг · ожидание · нужен ли VERIFY (code — всегда).
  - Источники гипотез: пороги гейтов (RSI-зоны, ADX≥18, `SCORE_MIN`), веса скор-семейств и потолки, `ATR_MULT`/`RR` по горизонтам, `QV_MIN`, дельты оверлеев BTC/новостей, вкл/выкл горизонтов и страт (журнал по стратам уже есть — убыточные страты режь первыми), механики выхода (трейлинг, partial TP, тайм-стоп), режимные фильтры.
- Держи план в контексте, работай **строго по нему**.

### 1. Этап ФУНДАМЕНТ (одноразовый — пока не построен, экспериментов НЕТ)
Раздай шаги сабагентам по плану (механика — sonnet, стратегия/бэктестер — opus):
- **Ф1. Вынести стратегию** из `gen-signals.mjs` в `scripts/strategy.mjs`: гейты, скор-семейства, оверлеи, entry/SL/TP, `evaluateSignal`/`closeSig`/`aggStats` — с **инъекцией параметров** (дефолты = текущие константы). `gen-signals.mjs` импортирует и ведёт себя **идентично**. Эквивалентность обязательна: прогон движка до/после на одном кэше свечей → одинаковый набор сигналов и метрик.
- **Ф2. Кэш истории** `scripts/fetch-history.mjs` → `scripts/backtest-data/` (**в .gitignore — репо публичный**): 1h/4h/1d/1w klines c `data-api.binance.vision`, **с quoteAssetVolume** (нужен для вселенной по датам); кандидаты — все USDT-пары `exchangeInfo`, глубина ≥ 12 мес 1h, ≥ 18 мес 4h, ≥ 24 мес 1d/1w; докачка инкрементальная.
- **Ф3. Бэктестер** `scripts/backtest.mjs` — walk-forward: на каждом шаге сигналы **только по закрытым свечам** (данные ≤ t), исход по свечам > t, двойное касание TP+SL в одной свече = **SL** (как в проде); вселенная топ-`TOP_N` **пересобирается на каждую дату** по дневному quoteVolume (анти-survivorship); дедуп открытых по ключу symbol+side+horizon как в проде; издержки прод-модели. Контракт CLI (пин для параллельных агентов): `node scripts/backtest.mjs --params <exp.json> --window train|val --out <res.json>` → netWinRate / avgNetR / medianNetR / totalNetPnlPct / maxDD_R / decided, всего и по стратам. Окна: train/val walk-forward, **≥ 2 непересекающихся val-окна, суммарно ≥ 25% истории**; разбиение фиксируется в STATUS и больше не двигается.
- **Ф4. Baseline:** прогон текущих дефолтов, метрики train и val → `docs/lab/STATUS.md` (структура: Фундамент-чеклист · Baseline · Текущие параметры · Журнал экспериментов · Бэклог гипотез · Известные смещения).
- **DoD фундамента:** двойной прогон бэктестера → бит-в-бит идентичный результат; look-ahead-тест (обрезка будущих свечей не меняет прошлые сигналы); Ф1-эквивалентность; smoke на 5 монетах. Коммит, draft PR, `Фундамент ✅ …`.

### 2. Этап ЭКСПЕРИМЕНТЫ — волны последовательно, пачки внутри волны ПАРАЛЛЕЛЬНО
Для каждой волны (≤ 3 волн за прогон):
- Выведи `Волна {…} взял.` Раздай пачки **одним сообщением, несколько Agent-вызовов параллельно**:
  - **`param`-пачка** (sonnet): агент пишет exp.json-варианты во временную папку, гоняет бэктестер по контракту Ф3 (train → отбор кандидата → val), **файлы репо не трогает**; возвращает компактно: цифры train/val по каждой гипотезе + рекомендацию.
  - **`code`-пачка** (opus, `isolation: worktree`): правит `strategy.mjs`/`backtest.mjs` в своём worktree, тот же протокол train→val, возвращает дифф + цифры. Параллельные code-пачки не пересекаются по файлам — иначе планировщик разносит их по волнам.
- **Гейт приёмки — на эксперимент, по val:** `totalNetPnlPct` > baseline **И** decided ≥ 200 (или ≥ `STRATUM_MIN` на каждую затронутую страту) **И** maxDD_R ≤ baseline × 1.2 **И** медиана netR не ниже baseline **И** **плато-чек**: ключевой параметр ±10% сохраняет ≥ 70% прироста. Любой провал → reject, в журнал с цифрами.
- **VERIFY-скептик на принятые** (один opus-агент на волну): опровергнуть — look-ahead, подгонка под val, survivorship, прирост на 2–3 сделках-выбросах, «улучшение» за счёт смягчения издержек/метрики, расхождение strategy.mjs↔бэктестер. Опроверг → reject с находкой.
- Принятые применяй **сам**: новые дефолты в `strategy.mjs` (прод-движок подхватывает), baseline ← новые цифры, журнал (id · гипотеза · train/val · вердикт · причина), коммит волны в ветку, PR обновляй. Выведи строку закрытия волны.
- **Стоп-правило:** две волны подряд без единого принятого → плато, завершай прогон.

### 3. Финал прогона
Прогони прод-движок разово (`pnpm signals`) — жив, схема `signals.json` обратно-совместима (фронт читает без правок). STATUS актуален, PR актуален. Выведи финальную строку с дельтой метрик. Затем — сводка (п.4).

### 4. Финальная сводка (в конце ЛЮБОГО прогона — и финал, и блокер-стоп)
Простым языком, без жаргона. Три части, пустую опускай: **Решения по ходу** (что выбрал · почему · альтернатива), **Хвосты** (бэклог гипотез, что взять следующим заходом, чем разблокировать блокеры), **На заметку** (неожиданности, риски, тяжёлые шаги). Плюс **обязательная строка честности**: цифры baseline→итог по val, число сделок, и оговорка — бэктест на истории, будущую доходность не гарантирует; известные смещения (делистнутые монеты недоступны и т.п.) — перечислить.

## Инварианты (анти-самообман — сетка не снимается)
- **Метрика неприкосновенна.** Издержки (`FEE_RT`/`SLIP_RT`/`SLIP_STOP_EXTRA`) двигаются только в сторону консервативности. «Прибыль выросла» из-за смягчения модели издержек или переопределения метрики — это reject, не результат.
- **Train/val — стена.** Гипотезы ищем и тюним на train; вердикт — по val; итерировать параметр против val нельзя (это тот же тюнинг на val).
- **Один источник логики.** `strategy.mjs` используют и прод-движок, и бэктестер. Разошлись — бэктест лжёт; чинить до любых экспериментов.
- **No look-ahead, исход пессимистично** (двойное касание = SL), вселенная — по датам, дедуп — как в проде.
- **Плато, не пик.** Хрупкий к ±10% параметр — подгонка, не сигнал.
- **Отрицательный результат — тоже результат:** в журнал с цифрами, не замалчивать.
- **Репо-гигиена:** `scripts/backtest-data/` в .gitignore; `news`/`arb`/PWA/`deploy.yml` не трогаем; работа в ветке `lab-*`, в `main` — только через PR (на `main` крон каждые 30 мин коммитит снапшоты — не мешаться с ним).
- Сомнение планировщика (тир/родство пачки/нужен ли VERIFY) → строже (`deep` / меньше пачка / VERIFY нужен).
- Контекст лёгкий: сабагенты возвращают сводки и цифры, не сырые логи бэктестов.
- Прервалось → продолжай по STATUS с первой незакрытой позиции (фундамент-чеклист или очередная волна).
